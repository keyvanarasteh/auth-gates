# Models & Configuration (`models.rs`)

`models.rs` dictates the data shapes expected by the analyzer and reports generated by the testing system.

## Core `TargetConfig` Struct
This structure encapsulates a full test campaign. It maps directly to what a user inputs.

```rust
pub struct TargetConfig {
    pub name: String,
    pub protocol: Protocol, // Http or WebSocket
    pub url: String,
    pub method: reqwest::Method,
    pub auth: AuthMethod, // Bearer, ApiKey, Basic, Cookie
    pub custom_headers: HashMap<String, String>,
    pub body: Option<String>,
    pub phases: Vec<PhaseConfig>,
    pub pre_test_login: Option<LoginConfig>,
    pub run_api_fuzzer: bool,
    pub run_ws_discovery: bool,
    pub fuzzer_wordlist: Option<Vec<String>>,
    pub generate_html_report: bool,
}
```

### Authentication Injection (`AuthMethod`)
- `AuthMethod::Bearer(token)` -> Generates an `Authorization: Bearer <token>` header.
- `AuthMethod::ApiKey(header, key)` -> Uses the specified header name to insert the `key`.
- `AuthMethod::Basic(user, pass)` -> Compiles HTTP Basic string (base64 encoded).
- `AuthMethod::Cookie(key, val)` -> Serializes to `Cookie: key=val`.

## Results
The `FinalReport` tracks aggregated statistics across the test suite:
- Total execution request volume
- Precise status code breakdown (`200`, `401`, `429`, `503` etc.)
- Whether a `429` (Rate Limit) was fired
- WAF classifications inside `security_headers`
- Serialized responses generated from the student-extracted features (API Fuzzers & WS Discovery)

This model is `Serialize + Deserialize` so it easily drops to disk via the `html_report.rs` structure.
